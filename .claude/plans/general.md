# General — deferred items / notes

## Deferred

- **Deals action buttons client-gate on `requireAuth` only** (`client/src/components/deals/DealsPageContent.tsx` — `onSubmitOffer`, `onRequestInfo`, and `requestDealInfo.onError`). The server routes require `requireSub(['basic','pro','premium'], { bypassRoles })`, so the client gate is looser than the server. Currently **latent**: the whole Deals page is fronted by `<AppAccessGate>`, so a no-access user can't reach those buttons. If the page-level gate is ever removed or the buttons surface elsewhere, tighten these to `requireSubscription` and surface the `403 Forbidden - Subscription required` instead of the generic error toast. (Noted 2026-06-16.)

- **Vendor mentions in Mastermind are HTML-embed only — not persisted to any table.** Tagging a vendor in a message stores the chip inside `messages.content` as `<span data-type="vendorMention" data-id="<vendor-uuid>" data-label="...">`; there is no DB row for it. The `message_mentions` table is **user-only** (drives notifications) and was deliberately left untouched — the server's mention parser is scoped to `data-type="mention"` spans, so vendor UUIDs are never recorded there and vendors get no notifications. Tradeoff: you currently **cannot** query "which messages mention vendor X" without scanning message content. **When we eventually attach users to vendors** (`vendors.userId` already exists but is nullable/unused for this) and want vendor tags to notify the linked account or power a vendor activity feed, that's the time to add persistence — the natural approach is extending `message_mentions` to hold both targets (make `mentioned_user_id` nullable, add `mentioned_vendor_id`, add a "exactly one of the two" check constraint) rather than a second table. (Noted 2026-06-17.)
